## 二、事故与技术根因（供后续规避）

1. **沙盒易失性 → 丢过两次工作**：Cowork 的 Linux 沙盒会被重置，未及时 commit 的改动全丢。
   根因：把工作长时间留在 /tmp 未推送。**教训：每完成一个可用改动立即 commit+push，不攒批。**
2. **主机/沙盒双文件系统错位**：用 Edit 工具写主机路径、用 python 写沙盒路径，两边不同步，提交了半套。
   根因：两套文件系统混用。**教训：单一路径来源，改完统一校验再提交。**
3. **SVG 画中文文字反复错位**：硬编码坐标 + 中文字宽不定 → 文字重叠/截断，观感"80年代"。
   根因：用错工具。**教训：带中文的图表一律 HTML/CSS（grid/flex）实现，不用 SVG text。**
4. **孤字占行反复出现**：窄单元格中文逐字换行。根因：缺 `word-break:keep-all`。已全局根治。
5. **证据分级差点被误删**：用户一度下令取消 ABC 分级（系统唯一质量闸门），我提示后果、他撤回。
   **教训：破坏性指令先讲后果、二次确认——此条已写入长期记忆。**
6. **权限/开关反复**：GitHub 细粒度令牌权限不足、Pages 开关被关，多次卡住需用户手动操作。

## 四、协作效率诊断

**高效的回合**：需求明确、是后端/数据/逻辑类、一次说清边界 → 我能一轮到位。
**低效的回合**：视觉迭代类、需求靠截图纠错、我看不到结果 → 多轮返工，是本项目时间的主要消耗。

**结论**：把工作按性质分流——**逻辑/数据/自动化交给我主导；视觉/审美由你主导决策、我执行，并务必给我"能看到结果"的通道（本地实时预览）。**

## 五、给后续 Claude 合作的操作性建议（最有价值部分）

1. **平台**：视觉迭代迁到 Claude Code + 本地 `python -m http.server` 实时预览。反馈闭环从"分钟级截图"降到"秒级刷新"，这是最大单点提效。
2. **提需求的方式**：一次给"目标+边界+反例"。例："关系图要保留可视化节点图（不是文字板），中文标签不重叠，参考 XX 风格"——比"这个不好看"能省 3 轮。
3. **可视化优先做 3 张**：关系图谱（节点图）、决策页主脉络流程图、先行引擎供应链图。用成熟库（Cytoscape/D3/Mermaid），别再手搓 SVG 坐标。
4. **决策报告是重点**：明确要"原创判断层"就单开一轮，只做这个；可接 LLM API 生成判读、输出标"AI 生成待核"，与机器模板层分离。
5. **每轮先出计划再动手**（你已要求过，继续保持）；破坏性变更等你明确授权。
6. **技术债清单**：index.html 单文件过大，建议在 Code 里拆成模块/组件；字体放大曾因沙盒重置丢失，需复核当前是否生效。
